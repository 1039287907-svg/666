# ISO 11820 三人分工方案

> 开发顺序：**Person A → Person B → Person C**（下层为上层提供接口）

---

## 一、整体依赖关系

```
┌─────────────────────────────────────────────────────────────┐
│  Person C — UI 层 + 导出层                                  │
│  LoginForm / MainForm / 子窗体 / ExportService              │
│  依赖：Person B 的事件 + Person A 的 DbHelper                │
├─────────────────────────────────────────────────────────────┤
│  Person B — 核心业务层                                      │
│  TestController / SensorSimulator / DaqWorker               │
│  依赖：Person A 的 DbHelper + Model + AppContext             │
├─────────────────────────────────────────────────────────────┤
│  Person A — 基础设施层                                      │
│  Model / DbHelper / AppContext / Config / Logging / DB初始化 │
│  依赖：无（最底层）                                          │
└─────────────────────────────────────────────────────────────┘
```

## 二、Person A — 基础设施 & 数据层（最底层，最先完成）

### 职责范围
数据库、配置、日志、全局上下文、所有数据模型。Person B 和 Person C 都需要调用你写的代码。

### 任务清单

#### 1. 数据模型类（Models 文件夹）
新建一个类库项目或 `Models` 文件夹，创建以下实体类（与 6 张表一一对应）：

| 类名 | 对应表 | 关键字段 |
|------|--------|---------|
| `Operator` | operators | userid, username, pwd, usertype |
| `Apparatus` | apparatus | apparatusid, innernumber, apparatusname, checkdatef, checkdatet, constpower |
| `ProductMaster` | productmaster | productid, productname, specific, diameter, height |
| `TestMaster` | testmaster | productid + testid（联合主键）, 以及文档中列出的全部字段 |
| `Sensor` | sensors | sensorid, sensorname, dispname, unit, outputvalue 等 |
| `CalibrationRecord` | CalibrationRecords | Id(GUID), CalibrationDate, TemperatureData(JSON), 9 测温点等 |

> 所有字段与 DB-数据库设计.md 中的建表 SQL 严格一致。`TestMaster` 字段最多（约 40 个），需逐一核对。

#### 2. DbHelper 类（数据库操作封装）
使用 `Microsoft.Data.Sqlite`，提供以下方法：

| 方法 | 说明 | 被谁调用 |
|------|------|---------|
| `Login(username, pwd)` → `(bool, userid, usertype)` | 登录验证 | Person C |
| `InitializeDatabase()` | 首次运行时建表 + 写初始数据 | 自己（启动时） |
| `UpsertProduct(ProductMaster)` | 插入或更新样品信息 | Person B / C |
| `GetAllProducts()` → `List<ProductMaster>` | 获取所有样品 | Person C |
| `GetApparatus()` → `Apparatus` | 获取设备信息（全局只有一条） | Person B / C |
| `UpdateApparatus(Apparatus)` | 更新设备信息 | Person C |
| `GetSensors()` → `List<Sensor>` | 获取所有传感器配置 | Person B |
| `UpdateSensorValue(sensorId, value)` | 更新单个传感器当前值 | Person B |
| `InsertTest(TestMaster)` | 新建试验记录（统计字段填 0） | Person B |
| `UpdateTestResult(productId, testId, ...)` | 试验完成时更新统计字段 + flag='10000000' | Person C |
| `QueryTests(from, to, productId?, operator?)` → `List<TestMaster>` | 按条件查询历史记录 | Person C |
| `GetCurrentTest()` → `TestMaster?` | 获取当前活动试验（flag != '10000000' 且 totaltesttime > 0 的最近一条） | Person B / C |

> **注意**：
> - 连接字符串由配置文件读取，不硬编码路径。
> - 所有 SQL 操作用参数化查询（`$param`），防止注入。

## 三、Person B — 核心业务层（中间层，依赖 Person A）

### 职责范围
试验状态机、仿真温度引擎、数据采集调度、温漂计算、试验终止判定。这是整个系统的"大脑"。

### 任务清单

#### 1. 事件参数类
先定义好与 UI 通信的事件数据结构：

```csharp
// 数据广播事件参数
public class DataBroadcastEventArgs : EventArgs
{
    public double TF1 { get; set; }        // 炉温1
    public double TF2 { get; set; }        // 炉温2

    public string StatusText { get; set; }  // 当前状态中文描述
    public double TemperatureDrift { get; set; } // 温漂
    public List<MasterMessage> Messages { get; set; } // 本轮消息
}

public class MasterMessage
{
    public string Time { get; set; }       // HH:mm:ss
    public string Message { get; set; }    // 消息内容
}
```

#### 2. 试验状态机（TestController）

定义枚举：
```csharp
public enum TestState { Idle, Preparing, Ready, Recording, Complete }
```

`TestController` 类需要实现：

| 成员 | 说明 |
|------|------|
| `CurrentState` | 当前状态（只读） |
| `IsStable` | 温度是否稳定（只读） |
| `ElapsedSeconds` | 记录阶段已过秒数（只读） |
| `event StateChanged` | 状态变化时触发 |
| `event DataBroadcast` | 每秒触发一次，携带所有实时数据 |
| `StartHeating()` | Idle → Preparing |
| `StopHeating()` | Preparing/Ready → Idle |
| `StartRecording()` | Ready → Recording |
| `StopRecording()` | Recording → Complete（需判断是否有有效记录） |
| `CreateNewTest(productId, testId, ...)` | 初始化新试验上下文 |
| `SaveRecord(postweight, phenocode, ...)` | 完成试验保存 |
| `SetupFixedDurationTest(productId, testId, seconds)` | 创建固定时长试验 |

**状态流转规则**（严格按开发文档 2.4 节）：
```
Idle ──[StartHeating]──→ Preparing
Preparing ──[温度达标+稳定]──→ Ready
Ready ──[StartRecording]──→ Recording
Recording ──[时间到/手动/终止条件]──→ Complete
Ready ──[温度跌落]──→ Preparing（回退）
Preparing/Ready ──[StopHeating]──→ Idle
Complete ──[SaveRecord]──→ Preparing（保持炉温等下次试验）
```

#### 3. 传感器仿真引擎（SensorSimulator）

创建 `SensorSimulator` 类，核心方法：`void Update(TestState state)`

严格按照开发文档 2.3 节的算法实现 5 通道仿真：

```
每一帧（800ms 调用一次）：

升温阶段（TF1 < 747°C）：
  TF1 += HeatingRate × 0.8 + Random(-1,1) × TempFluctuation
  TF2 += HeatingRate × 0.8 + Random(-1,1) × TempFluctuation  （与 TF1 独立噪声）
  TS  = TF1 × 0.3 + Random(-1,1) × TempFluctuation
  TC  = TF1 × 0.25 + Random(-1,1) × TempFluctuation
  TCal = TF1 + Random(-1,1) × TempFluctuation × 2

稳定阶段（TF1 >= 747°C）：
  TF1 = 750 + Random(-1,1) × TempFluctuation
  TF2 = 750 + Random(-1,1) × TempFluctuation
  稳定计数器++
  满足 745~755°C 且计数器 > 3 → IsStable = true

记录阶段（Recording）：
  surfaceTarget = min(TF1 × 0.95, 800)
  TS += (surfaceTarget - TS) × 0.02 + noise    // 指数接近
  centerTarget = min(TF1 × 0.85, 750)
  TC += (centerTarget - TC) × 0.01 + noise      // 比表面温更慢

降温阶段（停止升温后回到 Idle）：
  TF1 -= 0.5 + Random(-1,1) × TempFluctuation × 0.1
  TF2 -= 0.5 + Random(-1,1) × TempFluctuation × 0.1
```

> 所有配置参数从 `AppContext.Instance.Config.Simulation` 读取。

#### 4. 数据采集服务（DaqWorker）

使用 `System.Timers.Timer`，每 **800ms** 触发一次：

```
DaqWorker.Tick():
  1. 调用 SensorSimulator.Update(currentState)
  2. 将 5 通道当前值写回 AppContext.Instance.Db.UpdateSensorValue()
  3. 如果当前状态是 Recording：
     - ElapsedSeconds++
     - 将本秒的温度数据追加写入 CSV 文件（sensor_data.csv）
     - 将 PID 值加入恒功率队列
     - 检查终止条件
  4. 计算温漂（每 10 分钟数据做线性回归）
  5. 装配 DataBroadcastEventArgs
  6. 触发 DataBroadcast 事件
```

**恒功率计算**：
- Ready 状态下持续将 TF1 值记录到队列（最多 600 个）
- 用户点击"开始记录"时：恒功率 = 队列平均值

#### 5. 温漂计算

使用 `MathNet.Numerics.LinearRegression.SimpleRegression`：

```csharp
// 取最近 10 分钟（600 个数据点）的炉温1序列
// 做 y = a + b*x 线性回归
// 斜率 b 即为温漂（°C/10min）
double drift = regression.Slope() * 600;  // 换算为 °C/10min
```

#### 6. 试验终止条件

```
标准 60 分钟模式：
  - 在第 30/35/40/45/50/55 分钟检查
  - 炉温1和炉温2的 10 分钟温漂均不超过 MaxTemperatureDriftPerTenMinutes → 提前终止
  - 第 60 分钟无条件终止

固定时长模式：
  - 忽略提前终止检查
  - ElapsedSeconds >= TargetDurationSeconds → 终止

手动终止：
  - 用户点击"停止记录"
```

#### 7. CSV 写入

在 Recording 阶段，每秒追加一行到 `{BaseDirectory}\TestData\{productid}\{testid}\sensor_data.csv`：

```
Time,Temp1,Temp2,TempSurface,TempCenter,TempCalibration
```

文件首次创建时写表头。目录不存在时自动创建（`Directory.CreateDirectory`）。

## 五、三人的衔接接口（关键约定）

### A 提供给 B 的接口

```csharp
// AppContext 单例 — B 通过它获取一切
AppContext.Instance.Db          // DbHelper，所有数据库操作
AppContext.Instance.Config       // AppConfig，读取仿真参数等配置

// DbHelper 关键方法
DbHelper.GetSensors()           // → List<Sensor>，B 初始化传感器缓存
DbHelper.UpdateSensorValue(id, value)  // B 每帧更新
DbHelper.InsertTest(test)       // B 在新建试验时调用
DbHelper.GetCurrentTest()       // → TestMaster?，获取当前活动试验
DbHelper.GetApparatus()         // → Apparatus，获取设备参数
```

### B 提供给 C 的接口

```csharp
// TestController — C 通过它控制试验流程
TestController.CurrentState     // 用于按钮状态控制
TestController.StartHeating()   // 绑定"开始升温"按钮
TestController.StopHeating()    // 绑定"停止升温"按钮
TestController.StartRecording() // 绑定"开始记录"按钮
TestController.StopRecording()  // 绑定"停止记录"按钮
TestController.CreateNewTest(productId, testId, ...) // 新建试验后调用
TestController.StartFixedDurationTest(productId, testId, seconds) // 固定时长试验
TestController.SaveRecord(postweight, phenocode, ...) // 保存记录完成后调用

// DataBroadcast 事件 — C 订阅它来更新 UI

// DataBroadcastEventArgs 携带：
```

## 八、特别注意

1. **跨线程安全**：Person B 的 `DataBroadcast` 事件在后台线程触发，Person C 必须在回调中用 `this.Invoke()` 切换到 UI 线程。
2. **参数化 SQL**：Person A 的所有 SQL 必须用 `$param` 参数化，不要拼接字符串。
3. **CSV 目录创建**：Person B 写 CSV 前要 `Directory.CreateDirectory()` 确保路径存在。
4. **联合主键**：`testmaster` 的主键是 `(productid, testid)`，所有更新操作必须两个字段一起提供。
5. **flag 标记**：试验完成保存后必须设置 `flag='10000000'`，未保存的完成记录会阻止新建试验。
6. **OxyPlot 实时更新**：Person C 更新曲线数据后要调用 `PlotView.InvalidatePlot(true)` 刷新。
7. **EPPlus 许可**：EPPlus 7.x 在商业用途需要许可证，开发/学习用途可以设置 `ExcelPackage.LicenseContext = LicenseContext.NonCommercial`。
