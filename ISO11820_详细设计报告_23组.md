# 1 引言

## 1.1 编写目的

本文档是概要设计报告的细化和补充，对系统的数据模型、所有核心模块和关键算法进行详细设计描述，每个模块包含功能描述、输入、输出、处理逻辑和限制条件。本文档可直接指导编码实现。

## 1.2 设计范围

覆盖系统全部14个核心类的详细设计，包括6个数据模型类、1个数据库访问类、1个配置类、1个全局上下文类、1个状态机控制器、1个仿真引擎、1个数据采集服务、1个报告导出服务和4个窗体类，以及4个关键算法。

## 2.4 TestMaster（试验记录 — 核心实体）

**TestMaster包含约40个字段，按逻辑分组如下：**

【基本信息】productid(string PK+FK), testid(string PK, 格式yyyyMMdd-HHmmss), testdate(DateTime), ambtemp(double, 环境温度°C), ambhumi(double, 环境湿度%), according(string, 试验依据, 默认"ISO 11820:2022"), operator(string, 操作员用户名), apparatusid(string), apparatusname(string), apparatuschkdate(DateTime), rptno(string, 报告编号)。

【质量数据】preweight(double, 试验前质量g), postweight(double, 试验后质量g), lostweight(double, 失重量=pre-post), lostweight_per(double, 失重率%)。

【试验过程】totaltesttime(int, 总秒数), constpower(int, 恒功率值0~25600), phenocode(string, 现象编码), flametime(int, 火焰开始时刻秒), flameduration(int, 火焰持续时间秒)。

【温度最大值】maxtf1/maxtf2/maxts/maxtc(double, 各通道最大温度值) + 各自的_time(int, 最大值对应秒数)。

【温度最终值】finaltf1/finaltf2/finalts/finaltc(double, 各通道试验结束时刻温度值) + 各自的_time(int)。

【温升计算】deltatf1(double, 炉温1温升=finaltf1-初始值), deltatf2(double), deltatf(double, 判定项=当前取deltats), deltats(double, 表面温升), deltatc(double, 中心温升)。

【备注】memo(string, 可空), flag(string, 可空, "10000000"表示已保存完成)。

设计要点：新建试验时仅填写基本信息和preweight，其余统计字段全部填0；试验完成保存时执行UPDATE填充全部统计字段。

# 3 核心模块详细设计

## 3.1 DbHelper（数据库操作封装）

功能描述：封装所有SQLite数据库操作，提供参数化的CRUD方法。是整个系统对数据库的唯一访问入口。所有SQL语句使用$param参数化查询，防止SQL注入。连接字符串从配置文件读取，不硬编码。

关键方法：

| 方法签名 | 功能 | SQL操作 |
| --- | --- | --- |
| Login(username, pwd) → (bool, userid, usertype) | 登录验证 | SELECT userid,usertype FROM operators WHERE username=$name AND pwd=$pwd |
| InitializeDatabase() | 首次运行建表+初始数据 | CREATE TABLE IF NOT EXISTS ×6 + INSERT初始数据 |
| InsertTest(TestMaster) | 新建试验 | INSERT INTO testmaster (约40个字段) VALUES (参数化) |
| UpdateTestResult(productId, testId, ...) | 试验完成更新 | UPDATE testmaster SET postweight=..., flag="10000000" WHERE productid=$pid AND testid=$tid |
| QueryTests(from, to, productId?, operator?) → List | 多条件查询 | SELECT ... FROM testmaster WHERE testdate BETWEEN ... AND ... [AND productid LIKE ...] |
| GetSensors() → List<Sensor> | 获取全部传感器配置 | SELECT * FROM sensors |
| UpdateSensorValue(sensorId, value) | 更新传感器当前值 | UPDATE sensors SET outputvalue=$val WHERE sensorid=$id |
| UpsertProduct(ProductMaster) | 插入或更新样品 | INSERT OR REPLACE INTO productmaster ... |
| GetApparatus() → Apparatus | 获取设备信息 | SELECT * FROM apparatus LIMIT 1 |
| InsertCalibrationRecord(CalibrationRecord) | 保存校准记录 | INSERT INTO CalibrationRecords ... |
| QueryCalibrationRecords(...) | 查询校准历史 | SELECT * FROM CalibrationRecords ORDER BY CalibrationDate DESC |

性能考量：每个方法内部使用using语句确保SqliteConnection正确释放。SQLite不支持高并发写入（单写锁），但由于本系统是单用户单机应用，无并发问题。

限制条件：不使用ORM，直接编写SQL。表名和字段名需与数据库设计文档严格一致。CalibrationRecords表名首字母大写，与其他表不同，需注意。

## 3.2 AppConfig（配置管理）

功能描述：在程序启动时读取appsettings.json配置文件，将5个配置节的配置项暴露为强类型属性。使用Microsoft.Extensions.Configuration的ConfigurationBuilder模式。

配置节结构：

（1）DatabaseConfig：Provider(string), SqlitePath(string)

（2）HardwareConfig：ConstPower(int), PidTemperature(int), SensorProtocol(string)

（3）SimulationConfig：EnableSimulation(bool), SimulateSensors(bool), SimulatePidController(bool), InitialFurnaceTemp(double), TargetFurnaceTemp(double), HeatingRatePerSecond(double), TempFluctuation(double), StableThreshold(double), SimulateFlame(bool)

（4）FileStorageConfig：BaseDirectory(string), TestDataDirectory(string)

（5）ReportConfig：OutputDirectory(string), EnablePdfExport(bool)

实现要点：构造函数中通过IConfiguration.Bind()将JSON节点绑定到对应的配置对象。提供一个Reload()方法支持运行时重新读取配置。

## 3.3 AppContext（全局上下文单例）

功能描述：全局单例类，持有系统的核心对象引用（DbHelper、AppConfig、TestController），提供全局访问点。避免使用全局静态变量，便于单元测试时替换依赖。

公共属性：static AppContext Instance { get; }（单例）、DbHelper Db（数据库访问）、AppConfig Config（配置）、string CurrentOperator（当前登录用户名）、string CurrentUserType（当前角色类型）、TestController Controller（试验控制器，Person B完成后注入）。

限制条件：单例模式使用私有构造函数+静态只读字段实现（线程安全的懒加载）。所有属性在Program.cs启动流程中逐一注入。

## 3.4 TestController（试验状态机控制器）

功能描述：管理试验全生命周期的状态转换和业务规则。是系统的核心控制器，协调SensorSimulator、DaqWorker和数据库操作。

公共属性：TestState CurrentState(get; private set;)、bool IsStable(get; private set;)、int ElapsedSeconds(get; private set;)。

公共方法：

  StartHeating(): 检查CurrentState==Idle → 设置状态为Preparing → 触发StateChanged。

  StopHeating(): 检查CurrentState==Preparing||Ready → 设置状态为Idle → 触发StateChanged → 清空稳定计数器。

  StartRecording(): 检查CurrentState==Ready → 设置状态为Recording → 计算恒功率（队列平均值）→ 切换DaqWorker到记录模式 → 触发StateChanged。

  StopRecording(): 检查CurrentState==Recording → 检查是否有有效记录样本 → 设置状态为Complete → 触发StateChanged。

  CreateNewTest(productId, testId, ...): 初始化新试验上下文，重置所有计数器和缓存。

  SaveRecord(postweight, phenocode, flametime, flameduration, memo): 调用DbHelper.UpdateTestResult() → 调用ExportService生成报告 → 设置flag="10000000" → 状态回到Preparing。

事件：event EventHandler StateChanged、event EventHandler<DataBroadcastEventArgs> DataBroadcast。

程序逻辑：每个状态转换方法严格遵循"前置条件检查 → 执行状态变更 → 触发事件通知"的三步模式。前置条件不满足时静默返回，不抛异常。

限制条件：不支持并发试验（单炉设计）。状态不可跳过（如Idle→Recording被阻止）。Complete后不自动保存（需用户手动点击"试验记录"按钮）。

## 3.5 SensorSimulator（温度仿真引擎）

功能描述：根据当前试验状态和配置参数，每帧（800ms）生成一组5通道仿真温度数据。核心方法：void Update(TestState state)。

关键字段：double TF1, TF2, TS, TC, TCal（5通道当前温度值）、int stableCounter（稳定计数器）、Random rng（随机数生成器，实例级）。

输入：当前TestState、配置参数（TargetFurnaceTemp、HeatingRatePerSecond、TempFluctuation、StableThreshold）。

输出：更新5个double温度字段。

四大阶段处理逻辑详见第5.1节"温度仿真算法"。

性能要求：单次Update调用耗时 < 1ms。随机数使用System.Random（非线程安全，仅由DaqWorker线程调用）。

限制条件：温度值下限为0°C，无上限约束。随机噪声使用均匀分布Uniform(-1,1)。升温速率×0.8的原因是800ms=0.8秒。

## 3.6 DaqWorker（数据采集服务）

功能描述：系统的心跳，使用System.Timers.Timer以800ms为周期运行，在每个周期内完成仿真更新、数据库同步、CSV写入、终止检查和事件广播。

关键字段：Timer timer(AutoReset=true, Interval=800)、ConcurrentQueue<double> pidOutputQueue（恒功率计算队列，最多600个元素）、List<double> recentTemps（温漂计算用的最近温度序列）。

核心方法：void Timer_Elapsed(object sender, ElapsedEventArgs e)：

  （1）调用SensorSimulator.Update(currentState)

  （2）foreach传感器更新DbHelper.UpdateSensorValue(id, value)

  （3）if Recording: ElapsedSeconds++; 追加CSV行; pidOutputQueue.Enqueue(TF1); 调用CheckTermination()

  （4）recentTemps.Add(TF1); if recentTemps.Count > 600: recentTemps.RemoveAt(0)

  （5）if recentTemps.Count >= 600: 调用MathNet计算线性回归斜率 → drift = slope * 600

  （6）装配DataBroadcastEventArgs → 触发DataBroadcast事件

CSV写入逻辑：if (!File.Exists(path)): 写入表头行; File.AppendAllText(path, 数据行)。路径=BaseDirectory/TestData/{productid}/{testid}/sensor_data.csv。写入前调用Directory.CreateDirectory确保目录存在。

限制条件：仅支持仿真模式（EnableSimulation=true）。硬件模式需要额外的Modbus通信实现。温漂计算至少需要600个数据点（约10分钟），不足时返回double.NaN。

## 3.7 ExportService（报告导出服务）

功能描述：提供Excel和PDF两种格式的报告生成功能。读取CSV温度数据和testmaster数据库记录，生成标准格式的试验报告。

公共方法：

  static void ExportExcel(string testId, string csvPath, TestMaster test)：使用EPPlus生成.xlsx（3个Sheet）。

  static void ExportPdf(string testId, string csvPath, TestMaster test)：使用PDFsharp-MigraDoc生成.pdf。

Excel生成逻辑：详见概要设计4.4节。关键注意：EPPlus 7.x需设置LicenseContext=NonCommercial；Chart的数据源从CSV文件读取；Sheet3的曲线图使用LineChart类型。

PDF生成逻辑：详见概要设计4.4节。关键注意：中文字体需显式指定（如使用系统已安装的宋体/黑体）；OxyPlot导出位图使用PngExporter；判定结论使用MigraDoc的Paragraph写入。

限制条件：报告生成依赖于CSV文件完整性。如CSV文件缺失或为空，Excel的Sheet2和Sheet3将为空。PDF中的曲线图需要OxyPlot的PlotView实例进行截图导出。

# 4 界面模块详细设计

## 4.1 LoginForm（登录窗体）

功能描述：系统的入口窗体，提供角色选择和密码验证功能。

控件清单：两个RadioButton（rbAdmin="管理员", rbExperimenter="试验员"），一个TextBox（txtPassword, PasswordChar="*"），一个Button（btnLogin="登录"），一个Label（标题）。

输入：角色选择（RadioButton.Checked）、密码字符串（TextBox.Text）。

输出：登录成功 → 打开MainForm并关闭自身；登录失败 → MessageBox显示"密码错误，请重新输入"并清空密码框。

处理逻辑：btnLogin_Click事件：获取角色→映射用户名（admin→"admin", operator→"experimenter"）→调用AppContext.Instance.Db.Login(username, password)→（成功）设置AppContext.CurrentOperator和CurrentUserType→new MainForm().Show()→this.Hide()；（失败）MessageBox.Show("密码错误")→txtPassword.Clear()→txtPassword.Focus()。

限制条件：无用户名输入框（设计决策）。无登录失败次数限制（简化处理）。无"记住密码"功能。

## 4.2 MainForm（主窗体）

功能描述：系统的核心操作界面，集成试验操作、记录查询和设备校准三大功能区域。

控件清单（仅Tab1试验操作）：

  温度显示Panel：5个Label(lblTF1/lblTF2/lblTS/lblTC/lblTCal) LED大字体风格（黑底绿字，Font: Consolas 24pt Bold）

  信息显示Panel：lblStatus(状态文字+颜色)、lblTimer(已记录秒数)、lblDrift(温漂值)、lblProductId(样品编号)

  OxyPlot PlotView：4个LineSeries(红/蓝/绿/橙)，X轴LinearAxis(滚动最近10分钟)，Y轴LinearAxis(0~800)

  消息日志：RichTextBox(rtbLog, ReadOnly=true, BackColor=Black, ForeColor=White)

  操作按钮Panel：btnNewTest、btnStartHeat、btnStopHeat、btnStartRecord、btnStopRecord、btnRecord(试验记录)

Tab2和Tab3控件清单略（参见概要设计7.2节）。

输入：用户点击按钮的操作事件、DataBroadcast事件数据、StateChanged事件通知。

输出：更新的UI显示（温度值、曲线、消息、按钮状态）。

处理逻辑：

  MainForm_Load: 初始化OxyPlot→订阅TestController.DataBroadcast→订阅TestController.StateChanged→初始化按钮状态。

  OnDataBroadcast(sender, e): this.Invoke(() => { 更新5个温度Label.Text; 更新计时器; 更新状态标签+颜色; 更新温漂; 追加消息到rtbLog; 追加曲线数据点; plotView.InvalidatePlot(true); 调用UpdateButtonStates(); });

  OnStateChanged(sender, e): this.Invoke(() => { UpdateButtonStates(); });

  UpdateButtonStates(): 根据TestController.CurrentState和是否存在未保存记录来设置6个按钮的Enabled属性（参照状态控制矩阵）。

各按钮Click事件：调用对应的TestController方法（如btnStartHeat_Click→TestController.StartHeating()）。

限制条件：所有UI更新必须通过this.Invoke()切换到UI线程。OxyPlot更新后需调用InvalidatePlot(true)强制刷新。按钮状态更新逻辑较复杂，需同时考虑当前状态和未保存记录flag。

## 4.3 NewExperimentDialog（新建试验对话框）

功能描述：提供新建试验所需全部参数的录入界面。

控件清单：TextBox×4(txtProductId/txtProductName/txtSpecific/txtOperator(ReadOnly))、NumericUpDown×6(nudHeight/nudDiameter/nudPreWeight/nudAmbTemp/nudAmbHumi/nudCustomDuration)、RadioButton×2(rbStandard60Min/rbCustom)、Label×4(lblApparatusId/lblApparatusName/lblCheckDate/lblConstPower(自动填入))、Button×2(btnCreate/btnCancel)。

输入：用户填写的样品信息、环境参数、试验前质量、时长模式选择。

输出：调用DbHelper.InsertTest()和DbHelper.UpsertProduct()写入数据库，通知TestController.CreateNewTest()。

处理逻辑：btnCreate_Click：验证必填项(productid/productname/preweight>0)→生成testid=DateTime.Now.ToString("yyyyMMdd-HHmmss")→从AppContext获取当前操作员和设备信息→构造TestMaster对象(统计字段全部填0)→调用DbHelper.InsertTest()和UpsertProduct()→调用TestController.CreateNewTest()→DialogResult=OK→关闭。

限制条件：testid格式固定为yyyyMMdd-HHmmss，精度到秒。自定义时长输入框仅在rbCustom选中时显示和可用。设备信息为只读Label，不可编辑。

## 4.4 ExperimentRecordDialog（试验记录对话框）

功能描述：试验完成后填写火焰现象和试验后质量的对话框。

控件清单：CheckBox×1(chkFlame)、NumericUpDown×2(nudFlameTime/nudFlameDuration)、NumericUpDown×1(nudPostWeight)、TextBox×1(txtMemo, MultiLine)、Button×2(btnSave/btnCancel)。

输入：火焰现象（是否出现+时刻+持续时间）、试验后质量（必填）、备注（可选）。

输出：调用TestController.SaveRecord()，自动计算失重率、各通道温升、更新testmaster表，触发报告生成。

# 5 关键算法详细设计

## 5.1 温度仿真算法

算法目的：每800ms生成一组物理合理的5通道温度仿真数据。

算法输入：TestState currentState, double targetTemp(750°C), double heatRate(°C/s), double fluctuation(0.5), double stableThreshold(3°C)。

算法输出：double TF1, TF2, TS, TC, TCal。

伪代码：

  noise = () => (rng.NextDouble()*2 - 1) * fluctuation  // 均匀分布噪声

  if (state == Recording):

    // 表面温：以指数趋近模型向表面目标值逼近

    surfaceTarget = Min(TF1 * 0.95, 800)

    TS += (surfaceTarget - TS) * 0.02 + noise()

    // 中心温：以更慢的速率向中心目标值逼近

    centerTarget = Min(TF1 * 0.85, 750)

    TC += (centerTarget - TC) * 0.01 + noise()

  if (TF1 < targetTemp - stableThreshold):  // 升温阶段（< 747°C）

    TF1 += heatRate * 0.8 + noise()    // 0.8 = 800ms转秒

    TF2 += heatRate * 0.8 + noise()    // 独立噪声源

    if (state != Recording):

      TS = TF1 * 0.3 + noise()

      TC = TF1 * 0.25 + noise()

  else:  // 稳定阶段（>= 747°C）

    TF1 = targetTemp + noise()          // 直接钳位到目标温度

    TF2 = targetTemp + noise()

    stableCounter++

  TCal = TF1 + noise() * 2              // 校准温 = 炉温 + 双倍噪声

  if (state == Idle && 之前不是Idle):    // 降温阶段

    TF1 -= 0.5 + noise() * 0.1

    TF2 -= 0.5 + noise() * 0.1

算法复杂度：O(1)每帧，包含5次随机数生成和若干浮点运算。

## 5.2 温漂计算算法

算法目的：通过对最近10分钟炉温数据进行线性回归，计算温度漂移速率（°C/10min），用于温度稳定性判定和试验提前终止判定。

算法输入：List<double> recentTemps（最近600个炉温1数据点，800ms间隔，约10分钟）。

算法输出：double drift（°C/10min），数据不足600个时返回double.NaN。

伪代码：

  if (recentTemps.Count < 600): return double.NaN

  var regression = new SimpleRegression()

  for (int i = 0; i < recentTemps.Count; i++)

      regression.Add(i, recentTemps[i])   // (x=索引, y=温度值)

  double slope = regression.Slope()       // 每800ms的变温量

  double drift = slope * 600              // 换算为°C/10min

  return drift

使用MathNet.Numerics.LinearRegression.SimpleRegression类实现。斜率×600的换算逻辑：每个数据点间隔800ms，600个数据点对应480秒=8分钟，按比例外推到600秒=10分钟。

## 5.3 稳定判定算法

算法目的：自动判定炉温是否已达到目标温度并保持稳定，从而允许从Preparing切换到Ready。

判定条件（两个条件同时满足）：

  条件1：炉温1(TF1)在[TargetTemp - StableThreshold, TargetTemp + StableThreshold]范围内，即745°C~755°C。

  条件2：stableCounter > 3（连续3次以上tick满足条件1，即约3.2秒）。

伪代码：

  bool CheckStartCriteria():

    if (TF1 >= 745 && TF1 <= 755):

        stableCounter++

    else:

        stableCounter = 0

        IsStable = false

    if (stableCounter > 3):

        IsStable = true

        return true    // 触发 Preparing → Ready

    return false

## 5.4 终止判定算法

  CheckTermination():

    if (ElapsedSeconds >= 3600):  // 60分钟无条件终止
