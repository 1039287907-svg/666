# 1 引言

## 1.1 编写目的

本文档在需求分析报告的基础上，对ISO 11820建筑材料不燃性试验仿真系统进行概要设计。文档描述了系统的总体架构、模块划分、处理流程、数据结构和接口约定，为后续的详细设计和编码实现提供总体指导。

## 1.2 设计范围

本概要设计覆盖系统的全部软件组成部分，包括：Windows Forms界面层、业务核心逻辑层（试验状态机+温度仿真引擎+数据采集调度）、服务层（报告导出）、数据层（SQLite数据库访问）。不涉及硬件设计。

## 1.3 设计原则

（1）分层架构：严格遵循四层架构，上层依赖下层，下层不感知上层。

（2）事件驱动：后台线程与UI线程之间通过事件（event）通信，不直接调用UI方法。

（3）配置驱动：仿真参数、文件路径等可变参数集中在配置文件中管理。

（4）单一职责：每个类/模块负责一个明确的职责，降低耦合度。

（5）防御性设计：每个状态转换操作在执行前检查前置条件，防止非法状态变化。

## 3.2 模块间通信

模块间通信遵循以下约定：

（1）UI层 → 业务层：直接调用TestController的公共方法（如StartHeating()、StartRecording()等）。

（2）业务层 → UI层：通过事件（DataBroadcast事件）传递数据，UI层订阅事件并在Invoke中处理。

（3）所有层 → 数据层：通过AppContext.Instance.Db访问DbHelper的公共方法。

（4）所有层 → 配置：通过AppContext.Instance.Config读取AppConfig的属性。

（5）业务层 → 服务层：直接调用ExportService的静态方法。

## 3.3 模块依赖关系

依赖方向为自上而下：

Forms → Core → Data ↔ Models

Forms → Services → Data ↔ Models

Forms → Data (直接访问DbHelper进行查询操作)

所有模块 → AppContext (全局单例)

禁止的反向依赖：Data不能引用Core或Forms；Core不能引用Forms。

# 4 处理流程设计

## 4.1 系统启动流程

步骤1：Program.Main() 启动 → 配置Serilog日志 → 读取appsettings.json构建AppConfig → 创建AppContext单例并注入Config。

步骤2：创建DbHelper实例（传入配置中的数据库路径）→ 调用InitializeDatabase() → （a）首次运行：执行6张表的CREATE TABLE + 写入初始数据（2个操作员、1台设备、17个传感器通道） → （b）非首次运行：验证数据库完整性，跳过初始化。

步骤3：将DbHelper注入AppContext.Instance.Db → 启动LoginForm → Application.Run()。

## 4.3 数据采集流程

DaqWorker的数据采集循环（每800ms执行一次，由System.Timers.Timer驱动）：

（1）读取配置：检查EnableSimulation标志。仿真模式进入步骤2，硬件模式调用Modbus读取（当前版本仅实现仿真模式）。

（2）调用SensorSimulator.Update(currentState)：根据当前试验状态执行5通道温度仿真算法。

（3）同步数据库：将5通道当前温度值通过DbHelper.UpdateSensorValue()写入sensors表的outputvalue字段。

（4）Recording状态下的特殊处理：

    a. ElapsedSeconds自增。

    b. 将本秒的温度数据（Time+5通道值）追加写入CSV文件。

    c. 将TF1值加入恒功率队列（Ready状态开始积累，Recording状态开始使用队列平均值作为恒功率值）。

    d. 检查终止条件：标准60分钟模式在30~55分钟之间每5分钟检查温漂是否均低于阈值；固定时长模式比较ElapsedSeconds与目标时长。

（5）计算温漂：取最近600个炉温1数据点（约10分钟），使用MathNet.Numerics线性回归计算斜率，斜率×600得到温漂值（°C/10min）。

（6）装配并触发DataBroadcast事件：填充TF1/TF2/TS/TC/TCal温度值、ElapsedSeconds、StatusText、TemperatureDrift、Messages。

（7）触发StateChanged事件（如状态有变化）。

## 4.4 报告生成流程

ExportService的Excel报告生成流程：

（1）创建ExcelPackage → 设置LicenseContext为NonCommercial。

（2）Sheet1（试验信息表）：从testmaster表读取全部字段 → 格式化为两列表格（字段名|字段值）→ 底部添加判定结论行。

（3）Sheet2（温度数据）：读取CSV文件全部行 → 逐行写入Excel → 设置数值格式（温度保留1位小数）。

（4）Sheet3（温度曲线图）：使用EPPlus的Chart功能创建折线图 → 从CSV数据添加4个Series → 设置X轴（时间）和Y轴（温度0~800°C）→ 设置4色曲线。

（5）保存ExcelPackage到{OutputDirectory}/{testid}_报告.xlsx。

ExportService的PDF报告生成流程：

（1）创建MigraDoc Document对象 → 添加标题"ISO 11820 试验报告"。

（2）添加试验概要Section → 创建Table → 填充样品信息、试验参数、温升数据、失重率数据。

（3）将OxyPlot的PlotView导出为位图 → 嵌入PDF作为温度曲线图。

（4）添加判定结论段落 → 根据deltatf/lostweight_per/flameduration给出"通过"或"不通过"判定。

（5）使用PdfDocumentRenderer渲染并保存到{OutputDirectory}/{testid}_报告.pdf。

# 5 数据结构设计

## 5.1 数据库总体设计

使用SQLite关系型数据库，设计6张数据表：

| 表名 | 主键 | 记录数 | 关联关系 |
| --- | --- | --- | --- |
| operators | 无主键约束（实际用username唯一） | 2条（admin, experimenter） | 无外键 |
| apparatus | apparatusid (INTEGER) | 1条（FURNACE-01） | 无外键 |
| productmaster | productid (TEXT) | 随试验次数增长 | 被testmaster引用 |
| testmaster | (productid, testid) 联合主键 | 随试验次数增长 | productid→productmaster |
| sensors | sensorid (INTEGER) | 17条（0~16） | 无外键 |
| CalibrationRecords | Id (TEXT GUID) | 随校准次数增长 | ApparatusId→apparatus |

## 5.2 核心表结构（testmaster）

testmaster是系统的核心数据表，以(productid, testid)为联合主键，约40个字段分为7个逻辑分组：

（1）基本信息组（11字段）：productid, testid, testdate, ambtemp, ambhumi, according, operator, apparatusid, apparatusname, apparatuschkdate, rptno。

（2）质量数据组（4字段）：preweight, postweight, lostweight, lostweight_per。

（3）试验过程组（5字段）：totaltesttime, constpower, phenocode, flametime, flameduration。

（4）温度最大值组（8字段）：maxtf1/maxtf2/maxts/maxtc + 各自的_time。

（5）温度最终值组（8字段）：finaltf1/finaltf2/finalts/finaltc + 各自的_time。

（6）温升计算组（5字段）：deltatf1, deltatf2, deltatf, deltats, deltatc。

（7）备注组（2字段）：memo, flag。

设计索引：IX_Testmaster_Testdate（testdate）、IX_Testmaster_Operator（operator）、IX_Testmaster_Testdate_Productid（testdate, productid复合索引）。

# 8 错误处理设计

## 8.1 错误分类

| 错误类别 | 典型场景 | 严重级别 |
| --- | --- | --- |
| 数据库错误 | SQLite文件被占用/路径不可访问/磁盘空间不足 | 高 |
| 文件I/O错误 | CSV写入路径不可写/报告生成目录权限不足 | 中 |
| 配置错误 | appsettings.json格式错误/缺少必要配置项 | 高 |
| 跨线程操作 | 后台线程直接操作UI控件 | 高 |
| 业务逻辑异常 | 非法状态转换/数据完整性校验失败 | 中 |

## 8.2 处理策略

（1）数据库错误：使用try-catch包裹所有数据库操作，异常时通过Serilog记录详细错误信息（含SQL语句和参数），向用户显示友好的MessageBox错误提示，不暴露原始SQL异常信息。

（2）文件I/O错误：写入操作前检查目录是否存在（不存在则自动创建），写入失败时给出明确的路径和原因提示，不静默吞掉异常。

（3）配置错误：程序启动时验证必要配置项的完整性，配置项缺失或格式错误时给出明确提示并拒绝启动，不进入半初始化状态。

（4）跨线程操作：所有DataBroadcast事件回调中强制使用this.Invoke()切换到UI线程，编译期无法防止但运行时严格遵守编码规范。

（5）业务逻辑异常：状态转换方法在执行前检查前置条件（如StartRecording检查CurrentState==Ready），条件不满足时直接返回（静默拒绝），不改变当前状态。

# 9 系统部署设计

部署方式：绿色免安装，解压即用。

发布方式：使用dotnet publish -c Release --self-contained -r win-x64 生成自包含的单文件夹发布包，用户无需安装.NET运行时。

部署包文件清单：

├── ISO11820.App.exe                  [主程序]

├── appsettings.json                  [配置文件]

├── Data/

│   └── ISO11820.db                   [SQLite数据库文件（首次运行自动创建）]

├── *.dll                             [依赖DLL文件]

└── runtimes/                         [.NET运行时文件（self-contained模式）]

首次运行行为：检测数据库文件不存在 → 自动建表+写入初始数据 → 正常启动。后续运行：直接启动。
